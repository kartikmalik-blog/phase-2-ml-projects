WHAT IS THIS PROJECT?
---------------------
We built a Machine Learning model that predicts whether a
SpaceX Falcon 9 booster will successfully land after launch.
A successful landing saves approximately $60 million per
rocket by allowing reuse!

Dataset  : SpaceX Falcon 9 Launch Records
Launches : 90 real SpaceX missions
Features : 18 columns (launch site, orbit, payload, etc.)
Target   : Class (1=Success, 0=Failure)
Result   : 3 models achieved 100% Recall on failures!

KEY INSIGHT:
    Missing a failure prediction = $60M rocket crashes!
    False alarm = extra checks (cheap!)
    → RECALL for failures is our critical metric!
    Same thinking as fraud detection!

CELL 2 — LOADING & EXPLORING DATA
================================================================

THE DATASET FEATURES:
    FlightNumber  → Mission number (1, 2, 3...)
    Date          → Launch date
    BoosterVersion→ Falcon 9 version (Block 1-5)
    PayloadMass   → Cargo weight in kg
    Orbit         → Target orbit (LEO, GTO, ISS etc.)
    LaunchSite    → Where launched from (CCAFS, KSC etc.)
    Outcome       → Landing outcome description
    Flights       → How many times booster has flown
    GridFins      → Has grid fins? (True/False)
    Reused        → Is booster reused? (True/False)
    Legs          → Has landing legs? (True/False)
    LandingPad    → Which pad it's landing on
    Block         → Booster block version (1-5)
    ReusedCount   → How many times reused
    Serial        → Booster serial number
    Longitude     → Launch longitude
    Latitude      → Launch latitude
    Class         → TARGET: 1=Success, 0=Failure

TARGET BREAKDOWN:
    Success (1): 60 launches (66.7%)
    Failure (0): 30 launches (33.3%)
    Moderate imbalance → class_weight='balanced'!

KEY EDA FINDINGS:
    Success rate improved dramatically over years!
    Early launches (2010-2015) → mostly failures
    Recent launches (2018+) → mostly successes
    SpaceX got better with experience!

    Certain orbits have higher success rates
    LEO (Low Earth Orbit) → easier landing
    GTO (Geostationary) → harder, less fuel for landing

    Block version matters!
    Block 5 (latest) → highest success rate
    More mature technology → better performance

CELL 3 — FEATURE ENGINEERING
================================================================

WHY FEATURE ENGINEERING ON AEROSPACE DATA:
    Raw features like 'Date' and 'Reused' (True/False)
    need transformation to be useful for ML!
    Domain knowledge about rockets helps enormously!

ENGINEERED FEATURES EXPLAINED:

Year, Month, DayOfWeek
    → Extract from Date column
    → pd.to_datetime(df['Date']).dt.year
    → SpaceX improved over years → Year is powerful!
    → Launch scheduling patterns by month

IsWeekend
    → DayOfWeek >= 5 (Saturday or Sunday)
    → Weekend launches might have different conditions

IsReused
    → Convert boolean Reused to 0/1 integer
    → Reused boosters have different success rates!
    → .astype(int) converts True→1, False→0

HasGridFins
    → Convert boolean GridFins to 0/1
    → Grid fins help steer during landing
    → Missions with grid fins = better landing control!

HasLegs
    → Convert boolean Legs to 0/1
    → Landing legs required for successful touchdown!
    → Without legs = ocean splashdown (failure)

HeavyPayload
    → PayloadMass > 5000 kg → 1, else 0
    → Heavy payload = less fuel remaining for landing
    → Correlation with failures!

LightPayload
    → PayloadMass < 1000 kg → 1, else 0
    → Light payload = more fuel = easier landing

HighReuseCount
    → ReusedCount > 2 → 1, else 0
    → Experienced booster = better performance

IsFloridaLaunch
    → Latitude > 28 → 1, else 0
    → Florida launches = Cape Canaveral (CCAFS/KSC)
    → Different success rates by location

MatureBlock
    → Block >= 5 → 1, else 0
    → Block 5 = SpaceX's most refined design
    → Highest success rate of all versions!

CELL 4 — PIPELINE & TRAINING
================================================================

THE COLUMN TRANSFORMER:
    Different columns need different preprocessing!

    Numeric columns (Age, PayloadMass, etc.):
        → SimpleImputer(median) → fill missing
        → StandardScaler → scale to mean=0, std=1

    Categorical columns (Orbit, LaunchSite, etc.):
        → SimpleImputer(most_frequent) → fill missing
        → OneHotEncoder → convert text to numbers
           handle_unknown='ignore' → safe for new categories!

WHY DROP LEAKY COLUMNS?
    FlightNumber → just an ID, not predictive
    Date         → extracted features already created
    Outcome      → describes the landing result
                   THIS LEAKS THE ANSWER! Must remove!
    Serial       → unique booster ID, not generalizable

DATA LEAKAGE WARNING:
    'Outcome' column contains text like
    "True ASDS" (success) or "None ASDS" (failure)
    This directly reveals Class!
    ALWAYS drop columns that reveal the target!

PIPELINE ADVANTAGES:
    1. Consistent preprocessing for all models
    2. No data leakage in cross-validation
    3. GridSearch tunes model + preprocessing together
    4. Production-ready: one .predict() call does everything!

THE 5 MODELS + CLASS WEIGHT:
    All models use class_weight='balanced'
    → Compensates for 66/33 imbalance
    → Failure cases weighted 2x more than successes
    → Forces model to pay more attention to failures!

    XGBoost uses scale_pos_weight instead:
    → neg/pos = 60/30 = 2.0
    → Same effect as class_weight='balanced'


================================================================
CELL 5 — HYPERPARAMETER TUNING
================================================================

TUNING INSIDE PIPELINE:
    Special syntax for Pipeline parameter tuning:
    'model__n_estimators' (double underscore!)
    
    → 'model' = name of the Pipeline step
    → '__' = separator
    → 'n_estimators' = the actual parameter

    This is why Pipeline + GridSearch work together!
    GridSearch tunes the model parameters
    while Pipeline handles preprocessing correctly!

WHY cv=3 INSTEAD OF STRATIFIEDKFOLD:
    Small dataset (90 samples, only 18 test samples)
    StratifiedKFold can fail if folds too small
    cv=3 (integer) → sklearn handles it safely
    Simple is better for small datasets!

WHAT IS axes[] AND MATPLOTLIB?
================================================================

You asked why we use axes in plotting!
Here's a clear explanation:

BASIC PLOTTING:
    plt.figure() → creates a blank canvas
    plt.plot()   → draws on that canvas
    plt.show()   → displays it

MULTIPLE CHARTS (subplots):
    fig, axes = plt.subplots(2, 3, figsize=(18, 10))
    
    This creates a 2×3 GRID of charts!
    fig  = the entire figure (all 6 charts together)
    axes = array of individual chart objects
    
    axes[0][0] → top-left chart
    axes[0][1] → top-middle chart
    axes[0][2] → top-right chart
    axes[1][0] → bottom-left chart
    axes[1][1] → bottom-middle chart
    axes[1][2] → bottom-right chart

    axes[0][0].bar(...)  → draw bar chart in top-left
    axes[0][0].set_title('My Title')  → add title
    axes[0][0].set_xlabel('X Label')  → add x label

WHY USE axes INSTEAD OF plt?
    plt.plot() → draws on the CURRENT active chart
    axes[i].plot() → draws on a SPECIFIC chart
    
    When you have multiple subplots:
    plt methods get confused about which chart!
    axes[i] methods are explicit and clear!

SIMPLE RULE:
    Single chart → use plt.title(), plt.xlabel() etc.
    Multiple charts → use axes[i].set_title() etc.
